# 設計書

本ドキュメントは、OpenABEのアーキテクチャと設計原則について概説します。

## 1. システムアーキテクチャ
OpenABEは、数学的なプリミティブとハイレベルなアプリケーションロジックを分離するために、レイヤー化されたアーキテクチャを採用しています。

### レイヤーモデル
* **アプリケーション層**: CLIツールおよびサンプルアプリケーションで構成されます。この層はハイレベルAPI (ZCryptoBox) のみとやり取りします。
* **ラッパー層 (ZCryptoBox)**: 複雑な操作を簡素化するファサードです。鍵の状態管理、文字列からオブジェクトへの変換を行い、エンドユーザーに簡略化されたインターフェースを提供します。
* **コンテキスト層**: 特定の暗号スキーム（例：WatersによるCP-ABE）の実装です。この層で暗号化および復号の実際の数学的ロジックを処理します。
* **数学層 (ZML)**: 楕円曲線演算、双線形ペアリング、群要素を提供する基盤ライブラリです。ZMLはRELICやOpenSSLなどの高性能バックエンドのラッパーとして機能します。

## 2. 主要な設計パターン

### Strategy パターン
OpenABEは、複数のABEスキームをサポートするためにStrategyパターンを使用しています。`OpenABECryptoContext` はクライアントとして機能し、提供された設定に基づいて、実行時に異なるスキーム実装（例：Waters CP-ABE 対 GPSW KP-ABE）を切り替えることができます。

### Facade パターン
`ZCryptoBox` APIはファサードとして機能し、以下の複雑さを隠蔽します：
* システムパラメータの初期化。
* メモリ内での複雑な数学的オブジェクトの管理。
* 人間が読める形式のポリシーと内部表現との間のマッピング処理。

## 3. 実装されている暗号化パラダイム
OpenABEは、主に以下の3つのABEモデルを実装しています。

### 暗号文ポリシーABE (CP-ABE)
* **仕組み**: 暗号文にアクセスポリシーが紐付けられ、秘密鍵に属性が紐付けられます。
* **制御**: 暗号化者が、ユーザーの特性やロールに基づいて誰がデータを復号できるかを決定します。

### キーポリシーABE (KP-ABE)
* **仕組み**: 暗号文に属性が紐付けられ、秘密鍵にアクセスポリシーが含まれます。
* **制御**: 鍵発行者が、特定のポリシーに基づいてユーザーがどのデータを復号できるかを決定します。

### マルチオーソリティABE
* **仕組み**: 複数の独立したオーソリティが属性を発行でき、単一障害点や単一組織への完全な信頼を回避できます。

## 4. セキュリティ考慮事項
* **コリュージョン耐性 (Collusion Resistance)**: OpenABEは、ユーザーが各自の秘密鍵を組み合わせて、個々では満たせないポリシーを満足させることを防いでいます。
* **CCAセキュリティ**: ABE KEMの上に認証レイヤー（例：AES-GCM）を追加することで、選択暗号文攻撃 (CCA) への耐性を実装し、CPA安全なスキームをCCA安全なものに変換しています。
* **スレッド安全性**: スレッドセーフでない可能性がある基盤数学ライブラリのために、`OpenABEStateContext` を使用してスレッドごとの状態を管理しています。
